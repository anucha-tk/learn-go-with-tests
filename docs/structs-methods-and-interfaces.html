Structs, Methods & Interfaces

เรียนรู้วิธีการออกแบบประเภทข้อมูล (Types) ของเราเองในภาษา Go เพื่อซ่อนความซับซ้อนและเขียนโค้ดที่ยืดหยุ่นด้วย Interfaces ผ่านกระบวนการ TDD

เรื่องของเรา

สมมติว่าเราต้องการเขียนโปรแกรมทางเรขาคณิตเพื่อคำนวณหาเส้นรอบวง (Perimeter) และพื้นที่ (Area) ของรูปทรงต่าง ๆ ใน Go เราอาจเริ่มต้นด้วยการสร้างฟังก์ชันทั่วไปที่รับค่าดิบ เช่น float64 แต่เมื่อโปรแกรมโตขึ้น การส่งผ่านค่าดิบเหล่านี้ทำให้โค้ดสับสนและขาดความชัดเจนทางโครงสร้าง

ปัญหาแรกที่เจอ

เมื่อเราต้องการคำนวณพื้นที่ของวงกลม (Circle) เพิ่มเติมจากสี่เหลี่ยมผืนผ้า (Rectangle) การใช้ฟังก์ชันทั่วไปแบบปกติจะพบข้อจำกัด เพราะใน Go ไม่สามารถประกาศฟังก์ชันชื่อเดียวกันซ้ำใน package เดียวกันได้ (ไม่มีฟังก์ชันโอเวอร์โหลดเหมือนบางภาษา) ทำให้เราต้องคิดวิธีอื่นในการจัดการกับพฤติกรรม (Behavior) ที่ใช้ชื่อซ้ำกันนี้

ก่อนแก้ — แบบ naive

โค้ดเริ่มต้นที่ใช้ฟังก์ชันอิสระและรับตัวแปรดิบสำหรับคำนวณสี่เหลี่ยมผืนผ้า: 

func Perimeter(width float64, height float64) float64 {
    return 2 * (width + height)
}

func Area(width float64, height float64) float64 {
    return width * height
}

หลังแก้ — แบบ idiomatic

ประกาศโครงสร้างข้อมูลด้วย struct และเปลี่ยนมาใช้ method ที่ผูกติดอยู่กับโครงสร้างข้อมูลนั้น ๆ พร้อมกับใช้ interface เพื่อทำให้เขียนโค้ดรองรับรูปทรงใด ๆ ก็ได้ที่มีเมธอดที่กำหนด: 

package shapes

import "math"

// Shape กำหนดพฤติกรรมร่วมกัน
type Shape interface {
    Area() float64
}

// Rectangle สำหรับสี่เหลี่ยมผืนผ้า
type Rectangle struct {
    Width  float64
    Height float64
}

// Area ของ Rectangle
func (r Rectangle) Area() float64 {
    return r.Width * r.Height
}

// Circle สำหรับวงกลม
type Circle struct {
    Radius float64
}

// Area ของ Circle
func (c Circle) Area() float64 {
    return math.Pi * c.Radius * c.Radius
}

ทำไมต้องรู้เรื่องนี้

  • การจัดกลุ่มข้อมูล (Encapsulation): การใช้ struct ช่วยรวบรวมฟิลด์ที่เกี่ยวข้องกันและเพิ่มความชัดเจนให้กับเจตนาของโค้ด
  • การลดความผูกมัด (Decoupling): ใน Go การอ้างอิง interface เป็นแบบ implicit (ไม่ต้องการคีย์เวิร์ด implement) ขอเพียงแค่ Type นั้นมีเมธอดตรงตามรูปแบบ Interface ก็ผ่านการตรวจสอบประเภทข้อมูลทันที
  • ความสะดวกในการทดสอบ (Table-driven tests): การรวมเคสทดสอบด้วย interface ร่วมกับสไลซ์ของสตรักเจอร์นิรนาม ช่วยให้ขยายเคสทดสอบได้รวดเร็วและเป็นระบบ

หลุมที่เจอบ่อย

อาการ สาเหตุ วิธีแก้
Type does not implement Interface (missing Area method) โครงสร้างข้อมูลไม่มีเมธอดนั้น หรือลายเซ็น/ประเภทตัวแปรไม่ตรงกัน ตรวจสอบชื่อและลายเซ็นของเมธอดให้ตรงกับอินเตอร์เฟสอย่างถูกต้อง
เทสเฟลโดยระบุจุดผิดพลาดยากใน Table-driven tests ไม่ได้ใช้การรันแบบระบุชื่อย่อย (Sub-tests) ครอบแต่ละเคสด้วย t.Run(name, ...) และจัดรูปแบบข้อความด้วย %#v เพื่อให้รู้จุดผิดพลาดชัดเจน

สรุป

  • struct ใช้ประกาศชนิดข้อมูลขึ้นมาใหม่โดยรวมฟิลด์ต่าง ๆ ไว้ด้วยกัน
  • method คือฟังก์ชันพิเศษที่มี Receiver เชื่อมโยงกับประเภทข้อมูล
  • interface กำหนดเซตของเมธอดที่จำเป็น ทำให้เขียนโค้ดที่ยืดหยุ่นและลดการยึดติดของโค้ด (Decoupling)
  • Table-driven tests ช่วยให้เทสเคสครอบคลุมและบำรุงรักษาง่าย

ลองทำต่อ

  • ลองประกาศสตรักเจอร์ Triangle สำหรับสามเหลี่ยม แล้วเขียนเมธอด Area() เพื่อเพิ่มเข้าไปใน Table-driven tests
